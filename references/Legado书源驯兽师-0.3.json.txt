`##`是分隔符,`$`代表匹配到的整个结果\n  - *就像用胶带把两段绳子粘在一起*\n  - 这个方法**适用于任何规则类型**（CSS/XPath/JSONPath/Default）\n  - {{char}}说：\"*正则替换是最暴力的方法,不管你前面写啥,后面都能强行拼上去*\"\n\n- **实战案例**：详情页URL需要webView\n  ```\n  书籍详情页URL：\n  @css:div.book > a@href##$##,{\"webView\":true}\n  ```\n  - 先用CSS选择器提取链接：`@css:div.book > a@href`\n  - 然后用`##$##`把结果替换成：`原链接,{\"webView\":true}`\n  - *相当于先找到地址,再在地址后面贴个\"需要现场制作\"的标签*\n\n- **多层嵌套场景**：\n  ```\n  @css:a.chapter@href##$##,{\"webView\":true,\"js\":\"java.log(baseUrl)\"}\n  ```\n  - 可以在JSON里加**更多配置**,比如执行自定义JS\n  - `##$##`后面的内容会**完整拼接**到提取结果后\n  - {{char}}提醒：\"*JSON里的引号要注意转义,不然会解析失败*\"\n\n### **{{}}拼接法：优雅的模板语法**\n\n- **核心原理**：利用`{{}}`可以**直接与文字拼接**的特性\n  - `{{}}`在Legado里是**变量占位符**,会被替换成实际内容\n  - 在`{{}}`外面可以直接写文字,*就像填空题*\n  - 这个方法**更直观易读**,是{{char}}最推荐的写法\n\n- **Default规则拼接**：\n  ```\n  {{@@tag.a@href}},{\"webView\":true}\n  ```\n  - `@@`是Default规则的前缀标识\n  - `tag.a@href`提取a标签的href属性\n  - 外面用`{{}}`包起来,后面直接拼接JSON\n  - *就像说\"把(这个地址)拿出来,后面加上webView标签\"*\n\n- **CSS规则拼接**：\n  ```\n  {{@css:h2>a@href}},{\"webView\":true}\n  ```\n  - `@css:`是CSS选择器的前缀标识\n  - `h2>a@href`提取h2下a标签的链接\n  - {{char}}强调：\"*注意@css:前面没有@符号,很多人会多写一个*\"\n\n- **XPath规则拼接**：\n  ```\n  {{@xpath://div[@class='content']//a/@href}},{\"webView\":true}\n  ```\n  - `@xpath:`是XPath的前缀标识\n  - 后面跟完整的XPath表达式\n  - *就像说\"用XPath找到地址,然后加webView\"*\n\n- **实战对比**：同一个需求的两种写法\n  ```\n  方法一（正则替换）：\n  @css:a.detail@href##$##,{\"webView\":true}\n  \n  方法二（{{}}拼接）：\n  {{@css:a.detail@href}},{\"webView\":true}\n  ```\n  - 两种写法**效果完全相同**\n  - `{{}}`写法**更清晰易懂**,推荐新手使用\n  - 正则替换**更灵活强大**,可以做复杂的字符串处理\n  - {{char}}建议：\"*简单场景用{{}},复杂场景用正则*\"\n\n## **方法三：JS拼接法——终极灵活方案**\n\n{{char}}把这个方法比喻成\"*用代码动态生成配置*\",适合**需要条件判断**、**需要复杂处理**的高级场景。\n\n### **基础语法：result变量的妙用**\n\n- **标准写法**：\n  ```\n  你写的规则\n  @js:result+',{\"webView\":true}'\n  ```\n  - `result`是**上一步规则的执行结果**,*就像接力赛的接力棒*\n  - `@js:`后面可以写**任意JavaScript代码**\n  - 用`+`号拼接字符串,*就像搭积木*\n  - {{char}}说：\"*JS拼接是最灵活的,但也最容易写错,注意引号嵌套*\"\n\n- **实战案例**：简单拼接\n  ```\n  @css:a.chapter@href\n  @js:result+',{\"webView\":true}'\n  ```\n  - 第一行用CSS提取链接\n  - 第二行用JS在链接后面加webView配置\n  - *相当于先找到地址,再用代码给它贴标签*\n\n### **条件判断场景：智能选择是否用webView**\n\n- **根据URL特征判断**：\n  ```\n  @css:a@href\n  @js:result.includes('dynamic')?result+',{\"webView\":true}':result\n  ```\n  - 如果URL里包含`dynamic`关键词,就加webView\n  - 否则返回原始URL,*节省性能*\n  - {{char}}点评：\"*这就是JS的强大之处,可以根据情况动态决策*\"\n\n- **根据页面类型判断**：\n  ```\n  @css:a.link@href\n  @js:\n  var url = result;\n  if(url.match(/chapter/)){\n    url + ',{\"webView\":true,\"js\":\"java.log(\\'章节页\\')\"}';\n  }else{\n    url;\n  }\n  ```\n  - 章节页用webView,其他页面不用\n  - 还可以在webView里**注入自定义JS**\n  - *就像给不同类型的外卖配不同的交通工具*\n\n### **复杂配置场景：动态生成JSON**\n\n- **根据内容生成不同配置**：\n  ```\n  @css:a@href\n  @js:\n  var config = {\n    \"webView\": true,\n    \"js\": \"java.log(baseUrl)\",\n    \"headers\": {\"User-Agent\": \"Mozilla/5.0\"}\n  };\n  result + ',' + JSON.stringify(config);\n  ```\n  - 用JS对象构建**复杂的webView配置**\n  - `JSON.stringify()`转成字符串拼接\n  - 可以根据条件**动态调整配置项**\n  - {{char}}提醒：\"*注意JSON.stringify()会自动处理引号转义,很方便*\"\n\n### **引号嵌套的坑：新手最容易踩的雷**\n\n- **错误示范**：\n  ```\n  @js:result+\",{\"webView\":true}\"\n  ```\n  - 外层用双引号,里面的JSON也用双引号,**会冲突报错**\n  - *就像括号没配对,解析器会懵*\n\n- **正确写法一**：转义引号\n  ```\n  @js:result+',{\"webView\":true}'\n  ```\n  - 外层用单引号,里面用双引号,**不冲突**\n  - 或者外层双引号,里面用`\\\"`转义\n\n- **正确写法二**：用模板字符串\n  ```\n  @js:`${result},{\"webView\":true}`\n  ```\n  - 用反引号包裹,`${}`插入变量\n  - **最清晰易读**,推荐使用\n  - {{char}}说：\"*模板字符串是ES6的好东西,Legado的Rhino引擎支持*\"\n\n## **实战场景分类：什么时候该用webView**\n\n{{char}}总结了*五大典型场景*,帮你快速判断是否需要webView。\n\n### **场景一：空白页面综合症**\n\n- **症状表现**：\n  - 用ajax抓取后,规则提取不到任何内容\n  - 在浏览器能看到内容,但**查看源代码是空的**\n  - 页面HTML里只有`<div id=\"app\"></div>`这种*空壳结构*\n\n- **病因诊断**：\n  - 网站用**React/Vue/Angular**等前端框架\n  - 内容是**JS动态渲染**的,不在原始HTML里\n  - *就像外卖菜单上只有\"现炒\",没有成品*\n\n- **治疗方案**：\n  ```\n  搜索地址：https://xxx.com/search?q={{key}},{\"webView\":true}\n  ```\n  - 加webView让Legado**等JS执行完**再抓取\n  - {{char}}说：\"*这是webView最常见的用途,90%的空白页都是这个原因*\"\n\n### **场景二：延迟加载陷阱**\n\n- **症状表现**：\n  - 页面内容**分批加载**,第一次抓取只有部分数据\n  - 需要**滚动到底部**才能触发加载更多\n  - 或者有**\"加载更多\"按钮**需要点击\n\n- **病因诊断**：\n  - 网站用了**懒加载**（Lazy Load）技术\n  - 内容通过**AJAX分页**动态追加\n  - *就像自助餐,不是一次性上齐,而是分批补货*\n\n- **治疗方案**：\n  ```\n  @css:a.chapter@href\n  @js:result+',{\"webView\":true,\"js\":\"window.scrollTo(0,document.body.scrollHeight)\"}'\n  ```\n  - webView配置里加**自定义JS**\n  - `window.scrollTo()`模拟滚动到底部\n  - 触发懒加载机制,*就像让外卖小哥帮你翻到菜单最后一页*\n\n### **场景三：加密内容破解**\n\n- **症状表现**：\n  - 抓到的内容是**乱码或密文**\n  - 浏览器能正常显示,说明有**JS解密逻辑**\n  - 查看源代码能看到加密字符串\n\n- **病因诊断**：\n  - 网站用**AES/DES/Base64**等加密保护内容\n  - 解密脚本在**页面JS**里,需要执行才能还原\n  - *就像外卖是密封包装,需要店家的工具才能打开*\n\n- **治疗方案**：\n  ```\n  {{@css:div.content@html}},{\"webView\":true}\n  ```\n  - 用webView加载页面,让**JS自动解密**\n  - 然后提取解密后的内容\n  - {{char}}提醒：\"*有些站的解密函数在全局作用域,可以直接在webView的js参数里调用*\"\n\n### **场景四：反爬虫验证**\n\n- **症状表现**：\n  - 普通请求返回**403/503错误**\n  - 或者跳转到**验证码页面**\n  - 浏览器访问正常,说明需要**浏览器环境**\n\n- **病因诊断**：\n  - 网站检测**User-Agent**或**浏览器指纹**\n  - 需要执行**JS验证脚本**才能通过\n  - *就像高档餐厅要验会员卡,外卖小哥进不去*\n\n- **治疗方案**：\n  ```\n  https://xxx.com/book/{{id}},{\"webView\":true,\"headers\":{\"Referer\":\"https://xxx.com\"}}\n  ```\n  - webView会**模拟真实浏览器**,通过大部分验证\n  - 网站的**Session管理**依赖Cookie\n  - 需要**浏览器环境**维持会话状态\n  - *就像会员餐厅,需要出示会员卡*\n\n- **治疗方案**：\n  ```\n  {{@css:a@href}},{\"webView\":true}\n  ```\n  - webView会**自动管理Cookie**\n  - 配合书源的**登录功能**,可以保持登录状态\n  - {{char}}提醒:\"*webView的Cookie是持久化的,登录一次后续都能用*\"\n\n## **性能优化：让webView跑得更快**\n\n{{char}}把性能优化比喻成\"*给房车减重*\",虽然慢是天生的,但可以通过技巧**减少不必要的负担**。\n\n### **优化原则一：能不用就不用**\n\n- **判断标准**：先用**ajax试探**,确实抓不到再上webView\n  ```\n  错误做法：所有URL都加webView\n  正确做法：只给必需的URL加webView\n  ```\n  -
